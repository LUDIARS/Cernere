# Cernere セキュリティ設計書

## 1. 設計思想

Cernere は **常時接続セッションの強固な認証によって、外部からの破壊的変更を遮断する** ことを中核的なセキュリティ思想とする。

従来の REST API ベースの認証では、リクエスト単位でトークンを検証するステートレスなモデルが一般的である。しかし Cernere は、WebSocket による常時接続セッションを前提とし、セッションの存在そのものを継続的に検証することで、より強固な防御を実現する。

### 1.1 脅威モデル

| 脅威 | 対策 |
|---|---|
| 盗まれたトークンによる破壊的操作 | セッションの常時接続検証により、トークン単体では操作不可 |
| セッション乗っ取り | ping/pong による生存確認と接続の一意性保証 |
| 未認証リクエストの注入 | WebSocket 接続確立時の JWT/セッション ID 認証必須 |
| 他ユーザへの不正リレー | 同一ユーザのセッション間のみリレー許可 (最小権限) |
| 切断後の不正操作 | セッション切断時の即座な状態遷移 (`SessionExpired`) |

### 1.2 設計原則

1. **常時接続による継続的認証**: 一度の認証で終わりではなく、接続が維持されている間、30 秒間隔の ping/pong でセッションの正当性を継続的に検証する。
2. **破壊的操作のゲートキーパー**: データの削除・上書き・権限変更などの破壊的操作は、認証済みかつアクティブな常時接続セッションを経由しない限り実行されない。
3. **即座な無効化**: セッションが切断された場合、Redis 上のユーザステートを即座に `SessionExpired` に遷移させ、当該セッションからのすべての操作を拒否する。
4. **最小権限の原則**: リレーメッセージのスコープを同一ユーザのセッション間に制限し、権限境界を明確にする。

## 2. 認証アーキテクチャ

### 2.1 接続確立時の認証

```
クライアント                          Cernere
    │                                    │
    │─── GET /auth?token=<jwt> ──────────>│
    │                                    │── JWT 検証
    │                                    │── ユーザ存在確認
    │                                    │── SessionRegistry 登録
    │                                    │── Redis ustate 更新 (LoggedIn)
    │<── { type: "connected", ... } ────│
    │                                    │
```

- JWT トークンまたは既存セッション ID のいずれかが必須
- 未認証の場合は WebSocket アップグレードを拒否 (401)
- 認証成功時にセッション ID を発行し、SessionRegistry に登録

### 2.2 常時接続による継続的検証

```
    │<── { type: "ping", ts: ... } ─────│  (30秒間隔)
    │─── { type: "pong", ts: ... } ────>│
    │                                    │── 応答時間を記録
    │                                    │── タイムアウト検出
```

- サーバから 30 秒間隔で ping を送信
- クライアントは pong で応答する義務がある
- 一定時間応答がない場合、セッションを強制切断し `SessionExpired` に遷移

### 2.3 セッション切断時の即座な無効化

```
    │──× (接続切断) ×───────────────────│
    │                                    │── SessionRegistry から削除
    │                                    │── Redis ustate → SessionExpired
    │                                    │── 以降の操作を全拒否
```

## 3. 破壊的操作の防御フロー

### 3.1 破壊的操作の定義

以下の操作を「破壊的操作」として分類する:

| 操作 | エンドポイント | リスク |
|---|---|---|
| プロジェクト削除 | `DELETE /api/projects/{id}` | データ消失 |
| 設定値削除 | `DELETE /api/settings` | 設定消失 |
| プロジェクト上書き保存 | `POST /api/projects` | データ上書き |
| 設定値変更 | `POST /api/settings` | 設定変更 |

### 3.2 防御の多層構造

```
外部リクエスト
    │
    ├─ [Layer 1] セッション Cookie / Bearer Token 検証
    │   └─ 不正 → 401 Unauthorized
    │   実装: ws/auth.ts resolveWsAuth() / ws/project-handler.ts resolveProjectWsAuth()
    │
    ├─ [Layer 2] Redis セッション存在確認 (TTL 7日)
    │   └─ 期限切れ → 401 Unauthorized
    │   実装: commands.ts dispatch() の getUserState() 呼び出し
    │
    ├─ [Layer 3] ユーザステート検証 (LoggedIn であること)
    │   └─ SessionExpired → 403 Forbidden
    │   実装: commands.ts dispatch() の state !== "logged_in" 判定
    │
    └─ [Layer 4] 所有権確認 (リソースの所有者であること)
        └─ 他ユーザ → 403 Forbidden
        実装: 各 sub-dispatcher (requireSystemAdmin / 自データ判定 等)
```

### 3.3 プロジェクト WS の追加検証 (resolveProjectWsAuth)

`/ws/project` の upgrade 時、Layer 1+4 を統合した検証を行う:

1. JWT を HS256 で verify (Layer 1)
2. クレームの `clientId` で `managed_projects` を引く (実体確認)
3. `is_active = true` (廃止プロジェクト拒否)
4. クレームの `projectKey` と DB の `key` 列が一致 (token と DB の食い違い検知)

これにより、JWT が後から無効化された (プロジェクト削除/停止) ケースや、token と DB の整合性が崩れたケースを upgrade 段階で弾く。

### 3.4 監査ログの整合性

`commands.ts dispatch()` は finally 節で `operation_logs` に書き込む。書き込み失敗を silent catch で握り潰すと監査要件 (CLAUDE.md §1.2 Step 8) が崩れるため、**書き込み失敗時は `console.error` でメトリクス出力**する (運用側で監視する前提)。

## 4. リレーのセキュリティ境界

### 4.1 スコープ制限

- **デフォルト**: 同一ユーザの他セッションにのみリレー可能
- リレーメッセージの `target` フィールドで宛先を指定するが、サーバ側で所有権を検証
- 他ユーザへのリレーは Phase 2 で権限ベースの仕組みとして設計予定

### 4.2 リレーにおける破壊的操作

リレーメッセージのペイロードに含まれる操作も、受信側セッションの認証状態に依存する。未認証状態のセッションにリレーされたメッセージは、受信側で処理されない。

## 5. Redis ステート管理

### 5.1 ユーザステート遷移

```
None ──(認証成功)──> LoggedIn ──(切断/タイムアウト)──> SessionExpired ──(再認証)──> LoggedIn
                                                          │
                                                          └──(TTL超過)──> None
```

- `LoggedIn` 状態のみが操作を許可される
- `SessionExpired` への遷移は不可逆ではないが、再認証が必須
- Redis キー `ustate:{user_id}` に状態を保持

### 5.2 セッション TTL

- Redis セッションの TTL は 7 日間
- TTL 超過時にセッションは自動的に無効化される
- WebSocket 常時接続中は ping/pong により生存が保証される

## 6. まとめ

Cernere のセキュリティモデルは、以下の 3 つの柱で構成される:

1. **常時接続セッション**: WebSocket による持続的接続とその継続的検証
2. **強固な認証**: JWT/セッション ID による接続確立時の認証と、Redis によるステート管理
3. **破壊的操作の遮断**: 認証済みかつアクティブなセッション以外からの破壊的変更の完全なブロック

これにより、外部からの不正なリクエストがシステムに到達しても、正当なセッションを経由しない限り、データの破壊・改変は一切許可されない。
